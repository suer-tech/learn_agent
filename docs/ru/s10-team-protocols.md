# s10: Командные протоколы

`s01 > s02 > s03 > s04 > s05 > s06 | s07 > s08 > s09 > [ s10 ] s11 > s12`

> *"Тиммейтам нужны общие правила коммуникации"* -- один паттерн запрос-ответ управляет всеми переговорами.
>
> **Слой harness**: Протоколы -- структурированные рукопожатия между моделями.

## Проблема

В s09 тиммейты работают и общаются, но им не хватает структурированной координации:

**Завершение работы**: Убийство потока оставляет файлы недозаписанными, а `config.json` -- неактуальным. Нужно рукопожатие: ведущий запрашивает, тиммейт одобряет (заканчивает и выходит) или отклоняет (продолжает работать).

**Согласование плана**: Когда ведущий говорит «отрефактори модуль auth», тиммейт сразу приступает к делу. Для рискованных изменений ведущему стоит сначала пересмотреть план.

Оба протокола имеют одинаковую структуру: одна сторона отправляет запрос с уникальным ID, другая отвечает, ссылаясь на этот ID.

## Решение

Использовать один паттерн запрос-ответ для обоих рукопожатий: завершения работы и согласования плана. Инициатор генерирует уникальный `request_id`, отправляет сообщение через шину и держит запись в трекере со статусом `pending`. Получатель отвечает `approve` или `reject` со ссылкой на тот же `request_id`, и трекер переходит в `approved` либо `rejected`. Один FSM на `pending → approved | rejected` обслуживает любой такой протокол — добавление нового сводится к новому виду сообщения и новому трекеру.

## Как это работает

1. Ведущий инициирует завершение работы, генерируя `request_id` и отправляя его через входящие.

```python
shutdown_requests = {}

def handle_shutdown_request(teammate: str) -> str:
    req_id = str(uuid.uuid4())[:8]
    shutdown_requests[req_id] = {"target": teammate, "status": "pending"}
    BUS.send("lead", teammate, "Please shut down gracefully.",
             "shutdown_request", {"request_id": req_id})
    return f"Shutdown request {req_id} sent (status: pending)"
```

2. Тиммейт получает запрос и отвечает approve/reject.

```python
if tool_name == "shutdown_response":
    req_id = args["request_id"]
    approve = args["approve"]
    shutdown_requests[req_id]["status"] = "approved" if approve else "rejected"
    BUS.send(sender, "lead", args.get("reason", ""),
             "shutdown_response",
             {"request_id": req_id, "approve": approve})
```

3. Согласование плана работает по идентичному шаблону. Тиммейт отправляет план (генерируя `request_id`), ведущий рассматривает (ссылаясь на тот же `request_id`).

```python
plan_requests = {}

def handle_plan_review(request_id, approve, feedback=""):
    req = plan_requests[request_id]
    req["status"] = "approved" if approve else "rejected"
    BUS.send("lead", req["from"], feedback,
             "plan_approval_response",
             {"request_id": request_id, "approve": approve})
```

Один FSM, два применения. Один и тот же конечный автомат `pending -> approved | rejected` обслуживает любой протокол запрос-ответ.

## Что изменилось по сравнению с s09

| Компонент      | До (s09)         | После (s10)                  |
|----------------|------------------|------------------------------|
| Инструменты    | 9                | 12 (+shutdown_req/resp +plan)|
| Завершение работы | Только естественный выход | Рукопожатие запрос-ответ |
| Гейт планов    | Нет              | Отправка/рассмотрение с согласованием |
| Корреляция     | Нет              | `request_id` на каждый запрос |
| FSM            | Нет              | pending -> approved/rejected |
